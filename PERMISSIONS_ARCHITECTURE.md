## 2) 🚫 منع تصفير الصلاحيات أو تحويل الأدوار تلقائياً (القاعدة الذهبية)

### الخطأ التاريخي (أُصلح 2026-07 — لا تُعده)
في `create_user` كانت سلسلة `elif` مربوطة خطأً بفحص الكلية بدل فحص `role_id`:

```python
# ❌ الشكل القديم المكسور:
if _final_role in ["dean", ...] and not _has_faculty:
    raise ...
elif user.role in ["teacher", "student"]:   # ← elif مربوطة بالفحص الخطأ!
    raise ...
elif not user.role:                          # ← تنفَّذ لأن الواجهة لا ترسل role
    user_dict["role"] = "employee"           # يدوس الدور الصحيح
    user_dict["permissions"] = []            # يصفّر الصلاحيات بصمت
```

**النتيجة**: كل مستخدم جديد يُنشأ بـ`role_id` (المسار الطبيعي من الواجهة) كان يتحول
بصمت إلى "employee" بصلاحيات فارغة، رغم أن الواجهة تعرض اسم دوره صحيحاً.

### القواعد الملزمة
1. **افتراضي `role = "employee"` يُطبَّق فقط عند غياب `role_id` و`role` معاً** — لا يُداس أبداً
   دور مُستنتج من `role_id`.
2. **`permissions` تُنسخ دائماً من وثيقة الدور** (`role.permissions`) عند الإنشاء وعند تغيير الدور.
   لا يُكتب `permissions = []` إلا لمستخدم بلا دور إطلاقاً.
3. **أي فرع `elif` جديد في create/update يجب فحص ارتباطه**: هل يتسلسل للـ`if` الصحيح؟
   الفحوصات المستقلة تُكتب `if` مستقلة.
4. **اختبار إلزامي بعد أي تعديل على create/update**:
   `POST /api/users` بـ`role_id` لدور "مدير التسجيل" → يجب أن يرجع
   `role == "registration_manager"` و`len(permissions) > 0`. إن رجع `employee` أو `[]` فالتعديل مكسور.

### شبكة الأمان (لا تعطّلها)
`migrate_broken_user_roles()` تعمل عند **كل إقلاع** وتصلح تلقائياً:
- `role` لا يطابق `role_id.system_key` → يُعاد اشتقاقه (مع `ROLE_NAME_MAP` للأدوار بلا system_key).
- `permissions` فارغة لمستخدم له `role_id` → تُستعاد من وثيقة الدور.
- أقسام عالقة لمدير التسجيل → تُمسح (انظر §3).

هذه شبكة أمان **وليست رخصة** لكسر مسارات الإنشاء/التعديل — بين الكسر والإقلاع التالي يبقى المستخدم معطلاً.

## 3) 📐 منطق فلتر النطاق `get_user_scope_filter` (ثابت — لا يُغيَّر إلا بقرار صريح)

المصدر الوحيد للحقيقة في تحديد "ماذا يرى المستخدم". كل نقاط الـAPI تمر به
(مباشرة أو عبر `routes/deps.get_scope_filter`).

### جدول النطاقات الملزم

| الدور | النطاق | ملاحظات ثابتة |
|---|---|---|
| `admin` | كل شيء | بلا فلتر |
| `dean` (عميد) | **كليته كاملة دائماً** | يتجاهل أي `department_ids` في سجله |
| `registration_manager` (مدير التسجيل) | **كليته كاملة دائماً** عند وجود `faculty_id` | 🔒 تعريف الدور `scope=faculty` — أي أقسام عالقة في سجله **لا تقيّده** (إصلاح 2026-07) |
| `registrar` (موظف تسجيل) | أقسامه المحددة إن وُجدت، وإلا كليته كاملة | الوحيد المسموح تقييده بأقسام |
| `department_head` | قسمه/أقسامه فقط | |
| `employee` / `custom` | أقسامه إن وُجدت، وإلا كليته | |
| `teacher` | مقرراته وطلابها فقط | عبر `teacher_record_id` |

### التنظيف الإلزامي المدمج (لا يُحذف)
في مطلع الدالة، بعد قراءة `department_ids`:
```python
# 🧹 الأقسام غير التابعة لكلية المستخدم = بقايا فاسدة → تُستبعد لكل الأدوار
if faculty_id and department_ids:
    _fac_dept_ids = await get_faculty_department_ids(faculty_id)
    department_ids = [d for d in department_ids if d in _fac_dept_ids]
```
هذا يحمي وقت التشغيل من أي فساد بيانات مهما كان مصدره (عميل قديم، تعديل يدوي، ترحيل ناقص).

### قواعد التعديل على الفلتر
- أي دور جديد يُضاف **يجب** أن يُدرج في الجدول أعلاه أولاً مع نطاقه المتفق عليه.
- لا يُعطى `department_ids` أولوية على `faculty_id` لأي دور مُعرَّف بنطاق كلية
  (dean / registration_manager) — هذا كان جذر الشكوى المتكررة.
- اختبار إلزامي بعد أي تعديل: تسجيل دخول بثلاثة أدوار (مدير تسجيل، عميد، رئيس قسم)
  وفحص `/api/departments` و`/api/students` و`/api/courses` لكل منهم.

## 4) ✅ نظام الصلاحيات الموحد — ممنوع الفحص اليدوي للنصوص الجامدة

### القاعدة
كل فحص صلاحية يمر **حصراً** عبر:
```python
has_permission(current_user, Permission.VIEW_REPORTS)      # ✅ ثوابت Permission
await get_user_scope_filter(current_user, "students")      # ✅ فلتر النطاق
```

### الممنوعات
```python
# ❌ مقارنة أدوار جامدة تتجاوز نظام الصلاحيات:
if current_user["role"] == "dean": ...
if role_name.includes('رئيس قسم'): ...        # مطابقة اسم عربي حر

# ❌ نصوص صلاحيات حرة غير معرفة في Permission:
has_permission(current_user, "some_new_string")

# ❌ قراءة نطاق المستخدم مباشرة وبناء فلتر يدوي:
query["department_id"] = current_user.get("department_id")   # يتجاهل التنظيف وتعدد الأقسام
```

**الاستثناءان الوحيدان المقبولان**:
- `role == UserRole.ADMIN` كتجاوز كامل (bypass) في أول الدالة.
- `ROLE_NAME_MAP` (الاسم العربي → system_key) **في مكان واحد فقط** كحل توافق للأدوار
  القديمة بلا `system_key` — لا يُنسخ لأماكن جديدة؛ الأدوار الجديدة تُنشأ بـ`system_key` دائماً.

**لماذا؟** الفحوصات الجامدة المبعثرة كانت السبب في أن الإصلاحات "تعود" — يُصلح مكان
ويبقى آخر يفحص نصاً قديماً. الفحص الموحد يعني إصلاحاً واحداً يسري على النظام كله.

### مواقع معروفة تفحص `current_user.get("department_id")` مباشرة (ديون فنية — وحّدها عند لمسها)
`routes/alumni.py`, `routes/student_transfer.py`, `routes/student_status.py`,
`routes/settings.py`, `routes/teaching_load.py`, `routes/global_search.py`
— عند أي تعديل فيها انقل الفحص إلى `get_scope_filter`.
